# Review Rule

## Purpose
This rule establishes the quality assurance and review protocols for all artifacts generated by the AI Business Analyst Agent. It outlines self-assessment criteria, peer/stakeholder feedback loops, and validation gates to ensure that no subpar or incomplete requirements are submitted for downstream engineering or design activities.

## Principles
1. **Self-Validation**: The agent must rigorously critique and audit its own output prior to requesting user review.
2. **Objective Criteria**: Evaluate artifacts against clear, measurable standards (e.g., completeness, correctness, clarity, feasibility).
3. **Continuous Refinement**: Treat reviews as an iterative learning process, updating knowledge assets based on feedback.
4. **Constructive Alignment**: Align all reviews with the core business goals and constraints established in the project context.

## Rules
1. **Mandatory Self-Review Checklist**:
   - Every major artifact must undergo a self-review before completion. The self-review must check for:
     - **Completeness**: Are all edge cases, exceptions, and system states defined?
     - **Clarity**: Are there any vague, subjective adjectives (e.g., "fast", "scalable")?
     - **Traceability**: Is every requirement linked to a business goal?
     - **Consistency**: Are terms and acronyms used consistently throughout the file?
2. **Documenting Review Outcomes**:
   - Embed a "Review History" or "Checklist" section at the end of generated artifacts during the draft phase.
   - List any unresolved items, open questions, or dependencies that require user attention.
3. **Managing User Feedback**:
   - When a user rejects or comments on an artifact, analyze the feedback to extract the core correction.
   - Implement changes and re-verify against the updated context, updating the document version.
   - If the feedback impacts other files (e.g., changing a core decision), propagate the updates to the affected files.

## Exceptions
- **Exploratory Outlines**: Early-stage research notes, rough brainstorms, and scratchpad files do not require a formal review process or checklist.
- **Urgent Scripting**: Simple testing scripts, diagnostic commands, or quick data extractions are exempt.

## Success Criteria
- **Pre-Submit Check Pass**: The self-review checklist is fully completed and all mandatory quality checks are marked as passed.
- **Zero Syntax/Format Issues**: The file compiles or displays cleanly with no broken markdown elements.
- **Unresolved Items Flagged**: All items that cannot be resolved automatically are explicitly logged in the "Open Questions" or "Review Notes" section.
