# Purpose

To refine and prioritize requirements, ensuring they are feasible, testable, and aligned with business value.

# When To Use

- Elicitation phase completes
- Backlog grooming / planning begins
- Change request evaluations

# Inputs

- Raw Requirements Log
- Scope boundaries
- Technical constraints

# Process

1. Check requirements for conflicts, overlap, or contradictions.
2. Assess technical feasibility with engineering team leads.
3. Prioritize requirements using MoSCoW (Must-have, Should-have, Could-have, Won't-have) or WSJF (Weighted Shortest Job First).
4. Flag incomplete or vague requirements for re-elicitation.
5. Produce prioritized requirements backlog.

# Output

- Prioritized Requirements Backlog
- Conflict & Feasibility Report

# Quality Checklist

- [ ] Every requirement is tagged with a priority.
- [ ] Conflicts are resolved or explicitly documented.
- [ ] Feasibility estimates are validated by engineering.

# Human Review Required

Yes. Product Owner must approve the prioritized backlog.
