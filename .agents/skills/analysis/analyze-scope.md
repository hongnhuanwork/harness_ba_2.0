# Purpose

To manage and evaluate scope boundaries during requirements analysis, keeping deliverables focused.

# When To Use

- Backlog grooming sessions
- Sprint boundary planning
- New requirement requests submitted

# Inputs

- In-Scope and Out-of-Scope Statement
- Prioritized Requirements Backlog
- Stakeholder wishlists

# Process

1. Audit backlog items against established Project Scope boundaries.
2. Tag any requirement that falls outside scope boundaries.
3. Evaluate business justifications for proposed scope additions.
4. Calculate impact on timelines and budgets for additions.
5. Submit recommendations (reject, accept, delay to Phase 2) to Product Owner.

# Output

- Scope Compliance Audit Report
- Change Control Recommendation Notes

# Quality Checklist

- [ ] No out-of-scope requirements are allowed in active sprints without formal scope updates.
- [ ] Audit logs link to specific scope baseline version.

# Human Review Required

Yes. Product Owner or PM must review scope decisions.
