# Purpose

To evaluate the downstream impacts of proposed changes, preventing regression and unintended side effects.

# When To Use

- Change requests are submitted
- Scope modifications are proposed
- System upgrade analysis begins

# Inputs

- Change Request Forms
- Current state architecture maps
- Downstream system specifications

# Process

1. Map proposed change to active system components and workflows.
2. Identify affected modules, APIs, databases, and users.
3. Assess risks associated with data migration, system downtime, or training needs.
4. Estimate implementation effort, cost, and schedule impacts.
5. Summarize findings in an Impact Assessment Report.

# Output

- Impact Assessment Report
- Change Risk & Effort Estimate

# Quality Checklist

- [ ] All affected downstream systems are listed.
- [ ] Effort and timeline impact estimates are validated by Tech Leads.
- [ ] Mitigation strategies for regression risks are included.

# Human Review Required

Yes. Project Manager and Product Owner must approve the Impact Assessment.
