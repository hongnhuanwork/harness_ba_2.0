# Purpose

To prevent scope creep and ensure clear boundaries for the project team by explicitly documenting inclusions and exclusions.

# When To Use

- Project planning phase is active
- Scope questions or debates arise
- Preparation for Project Charter or BRD initialization

# Inputs

- Project Charter draft
- Stakeholder wishlists
- Technical constraints documentation

# Process

1. Analyze stakeholder expectations and initial business goals.
2. Group proposed features, systems, and processes into potential inclusions.
3. Identify and explicitly list items, systems, or processes that are out of scope for the current iteration.
4. Draft a Scope Statement outlining boundaries, constraints, and dependencies.
5. Review boundaries against time and budget constraints.

# Output

- In-Scope and Out-of-Scope Statement
- Boundary Definition Diagram / Matrix

# Quality Checklist

- [ ] Out-of-scope items are clearly and explicitly written to avoid ambiguity.
- [ ] In-scope items trace back to identified business goals.
- [ ] Scope boundaries match the project resources limits.

# Human Review Required

Yes. Formal scope boundaries must be approved by the steering committee or Product Owner.
