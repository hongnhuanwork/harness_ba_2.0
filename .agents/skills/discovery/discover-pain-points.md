# Purpose

To focus design and development efforts on solving the most critical operational and usability problems currently facing users.

# When To Use

- Discovery meetings or user interviews are conducted
- High escalation rates in legacy systems are observed
- Product roadmap planning is active

# Inputs

- User feedback logs and support tickets
- User interview transcripts
- System performance logs

# Process

1. Extract negative feedback, errors, delays, and workarounds from inputs.
2. Categorize pain points (e.g., Usability, Performance, Process, Data, Integration).
3. Estimate the business and operational impact of each pain point.
4. Map pain points to their root causes using 5-Whys or Fishbone analysis.
5. Prioritize pain points based on impact and resolution effort.

# Output

- Prioritized Pain Points Registry
- Root Cause Analysis Log

# Quality Checklist

- [ ] Every pain point is accompanied by evidence or user feedback quote.
- [ ] Root causes are distinguished from symptoms.
- [ ] Prioritization is based on objective metrics (e.g., hours lost, cost incurred).

# Human Review Required

No. Can be drafted independently but should be shared with the PO during backlog refinement.
