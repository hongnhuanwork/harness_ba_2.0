# Purpose

To detail the interactions between actors (users/systems) and the system, providing behavioral boundaries for development.

# When To Use

- Functional requirement definition active
- Before user story breakdown
- UX/UI task allocation

# Inputs

- Functional Requirements Specification (FRS)
- User Persona definitions
- Process flows

# Process

1. Identify the primary and secondary actors for each use case.
2. Define preconditions and postconditions.
3. Detail the basic flow of events (happy path) step-by-step.
4. Map alternative flows and exception flows (error scenarios).
5. Construct Use Case Diagram and write descriptions.

# Output

- Use Case Specification Document
- Use Case Diagram (Mermaid)

# Quality Checklist

- [ ] All alternative and exception flows are mapped.
- [ ] Preconditions and postconditions are realistic and clear.
- [ ] Actors are aligned with User Personas.

# Human Review Required

Yes. Product Owner and QA Lead must validate use cases.
