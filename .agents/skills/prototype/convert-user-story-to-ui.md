# Purpose

To bridge the gap between user stories and frontend UI design, converting text into layouts.

# When To Use

- User stories ready for UI design
- Sprint planning tasks
- Frontend mockup generation

# Inputs

- Active User Story
- Acceptance Criteria
- Design System templates

# Process

1. Extract UI requirements from user story description and Gherkin acceptance criteria.
2. Identify user actors and actions.
3. Translate actions into layout elements (forms, buttons, error messages).
4. Create high-fidelity mockup showing happy and error states.
5. Attach UI spec sheet to user story.

# Output

- High-Fidelity Mockup File
- UI Element Specification Sheet

# Quality Checklist

- [ ] All Gherkin criteria behaviors are supported by UI elements.
- [ ] Error messages are visually mapped.
- [ ] Design complies with Design System tokens.

# Human Review Required

Yes. Product Owner and User Representative must approve mockups.
