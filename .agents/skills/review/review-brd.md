# Purpose

To ensure the BRD satisfies quality gates before submission to client or development teams.

# When To Use

- BRD draft completes
- Client revision requests
- Pre-development audit

# Inputs

- Draft BRD
- Elicitation rules
- Project scope baseline

# Process

1. Audit the BRD structure against standard templates.
2. Verify that all requirements map to a business goal.
3. Scan for vague terminology or subjective criteria.
4. Ensure constraints, risks, and assumptions are fully documented.
5. Generate BRD Review Notes with recommendations.

# Output

- BRD Review Notes / Feedback Log

# Quality Checklist

- [ ] No sections are missing or contain placeholder content.
- [ ] Language is clear and contains no subjective adjectives.
- [ ] Document is compliant with GFM rules.

# Human Review Required

Yes. Product Owner or PM must review feedback before updating the BRD.
