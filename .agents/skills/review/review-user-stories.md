# Purpose

To ensure user stories are developer-ready, containing sufficient business rules, wireframes, and Gherkin criteria.

# When To Use

- Sprint refinement session
- DoR audit of backlog
- Pre-sprint planning

# Inputs

- User Stories drafts
- Definition of Ready checklist
- Gherkin criteria

# Process

1. Audit stories against INVEST criteria.
2. Verify that Gherkin Acceptance Criteria cover happy and exception paths.
3. Ensure that wireframes and user flow links are attached.
4. Flag stories that are too large (need epic split).
5. Update User Stories with review status.

# Output

- User Story Audit Log / Status updates

# Quality Checklist

- [ ] Acceptance criteria are complete and binary.
- [ ] No technical constraints are forced into story description.
- [ ] Estimate points can be reasonably assigned.

# Human Review Required

Yes. Product Owner and Tech Lead must validate stories.
