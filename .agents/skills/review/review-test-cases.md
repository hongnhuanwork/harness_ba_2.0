# Purpose

To verify that the QA test suite has complete coverage of all functional requirements and edge cases.

# When To Use

- QA Test Case catalog completes
- Sprint verification phase begins
- UAT preparation kickoff

# Inputs

- QA Test Cases
- Gherkin Acceptance Criteria
- RTM (Requirements Traceability Matrix)

# Process

1. Trace test cases back to requirements and user story criteria.
2. Verify that test data is realistic and defined.
3. Check that test steps are reproducible and expected results are verifiable.
4. Ensure negative testing and boundary testing are covered.
5. Draft Test Case Review Report.

# Output

- Test Case Review Report

# Quality Checklist

- [ ] 100% of acceptance criteria have matching test steps.
- [ ] No steps contain ambiguous instructions.
- [ ] Negative test paths are represented.

# Human Review Required

Yes. Lead QA Engineer must validate review outcomes.
