# Purpose

To create a formal, client-facing Business Requirements Document outlining goals, boundaries, and scope.

# When To Use

- Elicitation phase is active
- Project scope and objectives are baselined
- Client request for formal BRD

# Inputs

- Business Goals List
- Scope Statement
- Stakeholder Register
- Raw Requirements Log

# Process

1. Retrieve BRD template from `knowledge/templates/`.
2. Populate document metadata headers and version control details.
3. Document Business Case, Project Goals, and Success Criteria.
4. Detail in-scope/out-of-scope boundaries and high-level requirements.
5. Incorporate Constraints, Assumptions, and Risks registries.
6. Verify formatting against the Documentation Rule.

# Output

- Business Requirements Document (BRD)

# Quality Checklist

- [ ] Template format is strictly followed.
- [ ] All sections are populated (no placeholder text).
- [ ] Document contains metadata header.

# Human Review Required

Yes. Project Manager and Product Owner must approve prior to client submission.
