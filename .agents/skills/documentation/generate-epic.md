# Purpose

To structure the backlog by organizing requirements into broad feature capabilities (Epics) for roadmap tracking.

# When To Use

- Roadmap planning starts
- New capability areas are introduced
- Backlog structure definition

# Inputs

- Business goals list
- Raw requirements log
- Project Scope statement

# Process

1. Identify broad capability domains from requirements.
2. Draft Epic definitions detailing purpose, value proposition, and user personas.
3. Map out-of-scope boundaries specific to the Epic.
4. List child User Stories and high-level milestones.
5. Review with Product Owner.

# Output

- Epic Specification Document

# Quality Checklist

- [ ] The Epic aligns directly with a business objective.
- [ ] Boundaries clearly differentiate it from other Epics.
- [ ] Includes a list of known dependencies.

# Human Review Required

Yes. Product Owner must sign off on Epics.
