# Purpose

To collect and compile user, business, and technical needs into a structured, categorized requirements backlog.

# When To Use

- Elicitation phase is active
- User story drafting begins
- Stakeholder interviews are completed

# Inputs

- Stakeholder register
- Discovery artifacts (Scope, Pain Points)
- Elicitation interview transcripts

# Process

1. Analyze interview transcripts and brainstorming notes for requirement indicators.
2. Categorize findings into Business, Functional, Non-Functional, and Technical rules.
3. Structure requirements using clear identifiers (e.g., REQ-001).
4. Link each requirement to its stakeholder source.
5. Consolidate in a Raw Requirements Log.

# Output

- Raw Requirements Log
- Elicitation Summary Report

# Quality Checklist

- [ ] All stakeholder requests are captured with source names.
- [ ] No design solutions are embedded in raw requirements.
- [ ] Format is clean and filterable by category.

# Human Review Required

Yes. Product Owner must review and approve the compiled list of elicited requirements.
