# Purpose

To define exactly what the software must do in response to user inputs, triggers, and state changes.

# When To Use

- Functional specifications design begins
- User story map definition
- BRD to FRD conversion

# Inputs

- Raw Requirements Log
- User Persona definitions
- Use Case drafts

# Process

1. Extract requested system behaviors from raw elicitation materials.
2. Formulate detailed functional requirements specifying input, process, and output.
3. Format requirements clearly using active verbs (e.g., 'The system shall...').
4. Identify user actors and target systems for each function.
5. Log functional requirements with unique identifiers.

# Output

- Functional Requirements Specification (FRS)
- System Action-Response Table

# Quality Checklist

- [ ] Each requirement specifies actor, action, and expected system response.
- [ ] No design details or technologies are assumed in functional specs.
- [ ] Contains zero subjective adjectives (e.g., 'smoothly', 'quickly').

# Human Review Required

Yes. Product Owner and Tech Lead must validate functional capabilities.
